9. Keystone 하드웨어 지갑

Keystone 3 장치와의 통합. 소프트웨어 지갑과 전혀 다른 키 보관 모델과 서명 파이프라인을 쓰기 때문에 별도의 섹션으로 분리해서 설명한다.

9.1 키 보관 모델

Keystone 지갑은 소프트웨어 지갑과 달리 USK (Unified Spending Key) 가 phone 에 존재하지 않는다. 유저가 디바이스를 한 번 꺼내서 UFVK 만 phone 으로 복사하고, 서명은 매번 디바이스에서 수동으로 승인한다.

항목소프트웨어 계정Keystone 계정
Mnemonic Phone 의 secure storage (zcash_account_mnemonic_{uuid}) Phone 에 존재하지 않음 (디바이스에만)
USK 서명 직전 seed 에서 파생 → 서명 후 drop 디바이스 내부에만 존재, phone 은 절대 보지 않음
UFVK DB 에 저장 (zcash_client_sqlite) DB 에 저장 (QR/UR 로 import)
DB 상태 첫 계정: AccountSource::Derived 항상 AccountSource::Imported + AccountPurpose::Spending { derivation }
getActiveMnemonic() 니모닉 문자열 반환 null 반환
서명 경로 execute_proposal() 내에서 phone 이 서명 PCZT 를 device 에 넘겨서 device 가 서명

AccountProvider.isActiveAccountHardware 가 계정의 isHardware 플래그를 기반으로 send 경로를 분기한다 (software executeProposal vs hardware PCZT 파이프라인).

9.2 계정 Import — ZcashAccounts UR

Keystone 디바이스는 "Connect wallet" 메뉴에서 ZcashAccounts animated QR 을 띄운다. Phone 이 이걸 스캔하면 UR-encoded CBOR payload 가 구성된다:

UR:ZCASH-ACCOUNTS/...  (multi-part animated QR)
  └─ CBOR: ZcashAccounts
       ├─ seed_fingerprint: [u8; 32]
       └─ accounts: Vec<ZcashUnifiedFullViewingKey>
               ├─ ufvk: String (Orchard-only)
               ├─ index: u32 (ZIP 32 account index)
               └─ name: Option<String>

Orchard-only UFVK: 현재 Keystone 펌웨어는 UFVK 의 Sapling 부분을 export 하지 않는다. 그래서 import_hardware_account 는 기본 shielded_address_request (Orchard Require + Sapling Require) 대신 orchard_address_request (Orchard Require + Sapling Omit) 로 fallback 한다:

fn orchard_address_request() -> UnifiedAddressRequest {
    UnifiedAddressRequest::custom(
        ReceiverRequirement::Require, // Orchard
        ReceiverRequirement::Omit,    // Sapling — not in Keystone UFVK
        ReceiverRequirement::Omit,    // Transparent
    ).expect("valid receiver requirements")
}

결과: Keystone 계정의 수신 주소는 Orchard receiver 하나만 포함하는 UA. 유효한 Zcash 주소이지만 구형 Sapling-only 지갑한테서는 받을 수 없다.

9.3 Hardware-first 부트스트랩 제약

Keystone 계정은 지갑의 첫 계정이 될 수 없다. 빈 DB 에 Keystone import 를 시도하면 Dart 의 importKeystoneAccount 와 Rust 의 import_hardware_account 둘 다 거부한다.

이유: zcash_client_sqlite 의 init_wallet_db(Some(seed)) 는 DB 에 AccountSource::Derived 계정이 하나라도 있어야 seed 관련성 (relevance) 검증에 통과한다. Imported 계정만 있는 DB 는 seed-requiring migration 을 절대 적용할 수 없게 된다 — create_account 조차 사후 구제 경로가 아니다 (그 호출 자체가 seed-aware init 을 요구). 유일한 예방은 "첫 계정은 Derived" 불변 조건을 강제하는 것.

이 제약의 자세한 rationale 은 CLAUDE.md 의 "Multi-account migration limitation" 섹션 참고. 프로덕션 릴리즈 전에 이 tradeoff 는 재검토 예정.

9.4 Send Flow — Three-PCZT 파이프라인

하드웨어 서명은 3 개의 PCZT 를 사용하는 파이프라인이다. Keystone 디바이스는 ZK proof 를 생성할 수 없고 (proving key 가 너무 큼), phone 은 서명할 수 없으므로 (USK 가 디바이스에만 있음), 양쪽이 공통 base PCZT 의 별도 clone 에서 작업하고 phone 이 마지막에 합친다. zcash-android-wallet-sdk / Zashi 패턴과 동일하다.

1. createPcztFromProposal                      → base PCZT (phone)
   (IO-finalized, no proofs, no signatures)
      │
      ├── 2a. addProofsToPczt(base, params?)   → pcztWithProofs   (phone, CPU)
      │       Orchard proof 항상 생성; proposal.needsSaplingParams
      │       이면 LocalTxProver 로 Sapling output proof 도 생성
      │
      └── 2b. redactPcztForSigner(base)        → redactedPczt     (phone)
              Witnesses / proprietary metadata 제거 → QR frame 수 최소화
              → Keystone 디바이스 (animated QR or USB EAPDU)
              → 디바이스가 Orchard spend_auth_sig 서명
              → signed PCZT 가 phone 에                  → pcztWithSignatures
                                                       ↓
3. extractAndBroadcastPczt(
     pcztWithProofs, pcztWithSignatures,
     spend_params?, output_params?,
   )                                             → txid
     a. Combiner → TransactionExtractor (in-memory, DB 건드리지 않음)
     b. gRPC broadcast
     c. extract_and_store_transaction_from_pczt (primary, rich metadata)
        실패 시 → decrypt_and_store_transaction (fallback, correctness)
        둘 다 실패 시 → Err ("tx 는 네트워크에 있음")

9.6 Critical Invariants

각 항목은 개발 중에 실제로 터진 버그의 원인이다. 깨뜨리면 correctness 또는 data-loss 리그레션이 생긴다.

(1) extract_and_broadcast_pczt 는 반드시 broadcast 뒤에 persist

순서: TransactionExtractor::extract() (메모리 내, DB 건드리지 않음) → send_transaction gRPC → 그제서야 extract_and_store_transaction_from_pczt.

Store-then-broadcast 구조는 lightwalletd 가 tx 를 거절했을 때 복구 불가능한 상태를 만든다: DB 는 노트가 소비된 걸로 알고, 네트워크는 tx 를 본 적이 없고, 유저는 수동으로 지갑을 복구해야 한다.

(2) Broadcast 성공 후 local storage 실패는 send 실패로 surface 되면 안 됨

Primary: extract_and_store_transaction_from_pczt — PCZT recipient/memo metadata 를 보존한다 (history 에서 "sent to …" 표시).

Fallback: decrypt_and_store_transaction — sync 가 chain 에서 자기 자신의 sent tx 를 발견할 때 쓰는 경로와 동일. Correctness 는 보존되고 (nullifier 매칭으로 spent 노트 마킹), PCZT-only 표시 metadata 만 손실. 두 경로 모두 실패해야 에러가 surface 되며, 에러 메시지는 "tx 는 이미 네트워크에 있으니 재시도 말 것" 을 명시해야 한다.

(3) Sapling params 는 두 함수에 같은 쌍 으로 전달

PCZT 에 Sapling bundle 이 있으면 add_proofs_to_pczt 와 extract_and_broadcast_pczt 둘 다 Sapling params 가 필요하다. 한쪽은 LocalTxProver (proving), 한쪽은 LocalTxProver::verifying_keys() (extract + storage). send_screen.dart 가 같은 proposal.needsSaplingParams ? spendPath : null 를 두 FFI 호출에 각각 넘긴다.

한쪽만 넘기면 — 예를 들어 Sapling-only 주소로 송금 시도하면서 extract_and_broadcast_pczt 에만 None 을 준다면 — extraction 이 SaplingRequired 로 실패한다. 유저는 50MB params 다운로드와 Keystone 서명까지 다 하고서 암호 같은 에러만 본다.

(4) PROPOSAL_STORE 는 consume-on-entry + 취소 시 explicit discard

create_pczt_from_proposal 과 execute_proposal 모두 진입 시 .remove() 로 소비한다 (무거운 DB 작업 전에 store lock 을 drop). 같은 proposal_id 로 두 번째 호출 시 "Proposal not found (expired or already consumed)" 반환.

Dart _send() 는 전체 흐름을 try/finally 로 감싸고 proposalConsumed flag 가 consume 호출 직후 true 로 바뀐다. Finally 블록에서 flag 가 여전히 false 면 discardProposal(proposalId) 를 호출한다. 이로써 confirmation dialog 취소, Sapling params dialog 취소, Sapling 다운로드 중 예외, consume 호출 전의 어떤 에러도 전부 커버된다. discardProposal 은 idempotent.

9.7 UR (Uniform Resources) 스캐너

Animated QR 의 UR 프레임을 누적해서 완성된 CBOR payload 를 꺼내는 경로. rust/src/wallet/keystone.rs 에 구현.

  • Decoder: ur::Decoder 를 직접 사용 (KeystoneURDecoder 래퍼는 쓰지 않음 — URType::from() 레지스트리에 zcash-accounts 가 없어서 실패).
  • Global state: UR_SESSION: Mutex<Option<UrSession>>. ur_type 이 같은 타입의 in-flight 세션과 달라지면 auto-reset. 완료 시에도 auto-reset.
  • Reset entry: reset_ur_session() 은 #[flutter_rust_bridge::frb(sync)] 로 노출 — Dart scan screen 의 initState 에서 카메라 시작 전에 호출되어야 race 가 없다 (fire-and-forget Future 는 카메라 start latency 와 경쟁 가능).
  • Dart dedupe: _AnimatedUrScanScreenState._seenParts 는 decodeUrPart 성공 후에만 추가. Fountain-code 기반이라 transient 실패한 fragment 는 다음 animation cycle 에서 자동으로 다시 들어옴.
  • Post-await guard: _onDetect 는 await 뒤에 if (!mounted || _complete) return 로 late arrival 을 버린다. 동시 frame 처리나 scan 중간 dispose 에서 setState 가 disposed widget 에 호출되는 race 방지.

9.9 USB 경로에 대한 주의

현재 rust/src/wallet/keystone.rs 에 KEYSTONE_VID / KEYSTONE_PID 기반 EAPDU USB 프로토콜 코드가 살아있지만, Keystone 3 펌웨어가 Zcash 를 USB 로 지원하지 않기 때문에 런타임 경로는 작동하지 않는다. 코드는 펌웨어 지원이 추가되었을 때 다시 활성화할 수 있도록 보존되어 있다. 프로덕션 릴리즈 전에 이 상태를 재평가할 예정 (UI 에서 가릴지, 플래그로 게이트할지, 코드를 제거할지).

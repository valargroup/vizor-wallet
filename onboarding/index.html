1.3 왜 Rust + Flutter인가

공식 Zcash 생태계에서 Swift(iOS)와 Kotlin(Android) 모바일 SDK를 제공하지만, 둘 다 내부적으로 동일한 librustzcash Rust 크레이트를 UniFFI / JNI로 래핑한 것이다. 공식 Flutter/Dart SDK는 존재하지 않는다.

우리 접근법은 librustzcash를 FFI로 직접 호출하며, 이를 통해:

  • 단일 Rust 코드베이스로 iOS와 Android 모두 커버 — 플랫폼별 암호화 코드 불필요
  • zcash_client_backend와 zcash_client_sqlite에 중간 SDK 없이 직접 접근
  • flutter_rust_bridge가 크로스 컴파일, async, 타입 마샬링을 자동으로 처리

2. Flutter Rust Bridge (FRB) 통신

2.1 FRB 작동 방식

FRB는 rust/src/api/의 Rust 소스 파일을 읽고 Dart 바인딩을 생성한다. 런타임에:

  1. Dart가 생성된 async 함수를 호출 (예: createWallet())
  2. FRB가 SSE(Simple Serialization Engine)로 인자를 직렬화
  3. 별도 isolate/thread의 Rust 함수로 호출이 전달됨
  4. Rust가 함수를 실행하고 결과를 직렬화
  5. FRB가 결과를 역직렬화하여 Dart 객체로 반환

#[flutter_rust_bridge::frb(sync)]로 표시된 Rust 함수는 호출 스레드에서 동기적으로 실행된다(isolate 전환 없음). 간단하고 블로킹하지 않는 연산에만 사용해야 한다.

2.3 Codegen 워크플로우

# rust/src/api/*.rs 파일을 수정한 후 (pub fn 시그니처나 pub struct 변경 시):
flutter_rust_bridge_codegen generate

# 재생성되는 파일:
#   lib/src/rust/api/*.dart        (Dart 바인딩)
#   lib/src/rust/frb_generated.*   (FFI 글루 코드)
#   rust/src/frb_generated.rs      (Rust FFI 글루 코드)

중요: codegen은 프로젝트 루트(/zcash-wallet/)에서 실행해야 한다. rust/에서 실행하면 안 된다. flutter_rust_bridge.yaml 설정이 프로젝트 루트에 있기 때문이다.

2.4 설계 제약

  • API 경계에는 단순 타입만 사용. FRB는 프리미티브, String, Vec, 플랫 struct를 잘 처리한다. 복잡한 Zcash 타입(WalletDb, UnifiedSpendingKey, CompactBlock)은 Rust 내부에 머물러야 한다. API 레이어(rust/src/api/)가 단순 타입과 내부 Zcash 타입 간의 변환을 담당한다.
  • 호출 간 Rust 상태를 유지하지 않는다. 각 FFI 호출은 DB를 열고, 연산을 수행하고, 닫는다. Rust에 장기 유지되는 WalletDb 인스턴스가 없다. 더 단순하고 라이프사이클 문제를 피할 수 있다.
  • 에러는 문자열이다. API 전체에서 Result<T, String>을 사용한다. Zcash crate 에러는 API 경계에서 사람이 읽을 수 있는 문자열로 포맷된다.

3. Rust 핵심 로직

3.1 모듈 구조

rust/src/

                          # get_sync_status(), get_balance(), send, history 등
                          # (개별 sync 단계 함수는 legacy dead code)
    simple.rs             # FRB 데모 (greet 함수, 제거 가능)
  wallet/

    sync.rs               # 잔액 조회, send, history, address validation 등

                          # MemoryBlockSource (BlockSource trait 구현)

api/ 레이어가 FFI 경계이며, 파라미터를 파싱하고 wallet/에 위임하는 얇은 래퍼만 포함한다. wallet/ 레이어에 실제 Zcash 로직이 들어있다.

두 가지 FFI 경로: Dart는 FRB를 통해 api/sync.rs의 start_full_sync()를 호출한다. iOS background에서는 Swift가 C FFI를 통해 ffi.rs의 zcash_run_full_sync()를 호출한다. 둘 다 내부적으로 sync_engine::run_sync_inner()를 실행한다.

ffi.rs가 api/ 밖에 있는 이유: FRB codegen이 api/ 하위의 모든 pub fn을 Dart 바인딩으로 변환하는데, C FFI 함수(extern "C", *const c_char 등)는 FRB가 처리할 수 없다.

3.2 Wallet API — 키 유도 & 계정 관리

파일: rust/src/wallet/keys.rs

니모닉 생성: bip0039 크레이트로 256비트 엔트로피의 24단어 BIP-39 니모닉을 생성한다. Seed 유도는 표준을 따른다: PBKDF2-HMAC-SHA512, salt = "mnemonic" + passphrase (빈 passphrase).

키 유도: zcash_keys::UnifiedSpendingKey::from_seed()가 하나의 seed에서 세 종류의 키를 유도한다:

  • Transparent: BIP-44 경로 m/44'/133'/0' (mainnet)
  • Sapling: Jubjub 곡선 위의 ZIP-32 HD 유도
  • Orchard: Pallas 곡선 위의 ZIP-32 HD 유도

계정 생성: WalletDb::create_account()은 Unified Full Viewing Key (UFVK)만 데이터베이스에 저장한다. Spending key는 DB에 영속화하지 않는다.

주소 인코딩: Unified Address는 Require Orchard + Require Sapling + Omit Transparent로 생성하며, zodl/Zashi 월렛 동작과 일치한다. shielded_address_request() 헬퍼 함수가 이를 제어한다.

fn shielded_address_request() -> UnifiedAddressRequest {
    UnifiedAddressRequest::custom(
        ReceiverRequirement::Require, // Orchard
        ReceiverRequirement::Require, // Sapling
        ReceiverRequirement::Omit,    // Transparent
    ).expect("valid receiver requirements")
}

3.3 Sync API — 블록 스캐닝 & 잔액

파일: rust/src/wallet/sync.rs, rust/src/wallet/sync_engine.rs

Sync는 Rust에서 통합 실행된다. sync_engine.rs의 run_sync_inner()가 gRPC 연결, 블록 다운로드, 스캔, enhancement를 모두 처리한다. Dart는 start_full_sync() 하나만 호출하고, 진행률은 get_sync_status()와 get_balance()를 2초마다 폴링하여 확인한다.

3.4 Zcash Crate 의존성 그래프

zcash_client_sqlite (0.19.5) <-- 우리의 주요 진입점 |-- zcash_client_backend (0.21.2) | |-- zcash_keys (0.12.0) | |-- zcash_protocol (0.7.2) | |-- zcash_address (0.10.1) | |-- zcash_primitives (0.26.4) | +-- lightwalletd proto (내부 tonic gRPC) |-- sapling-crypto (0.5.x) <-- zcash_client_sqlite의 버전과 반드시 일치해야 함 +-- orchard (0.11.x) <-- zcash_client_sqlite의 버전과 반드시 일치해야 함 tonic (0.14) <-- gRPC 클라이언트 (tls-ring + tls-webpki-roots)

버전 제약: orchard와 sapling-crypto 버전은 zcash_client_sqlite가 내부적으로 사용하는 것과 반드시 일치해야 한다. 다른 버전을 사용하면 타입 불일치 에러가 발생한다 (예: "expected sapling_crypto::Node, found Node"). Rust는 서로 다른 crate 버전을 다른 타입으로 취급하기 때문이다.

3.5 보안 모델: Spending Key 분리

이것은 librustzcash의 의도적인 설계이며, 우리의 선택이 아니다:

  • WalletDb에는 Unified Full Viewing Key (UFVK)만 저장된다 — 트랜잭션 감지와 잔액 계산에는 충분하지만, 자금을 사용할 수는 없다.
  • 니모닉 / seed는 Dart의 flutter_secure_storage(iOS Keychain / Android Keystore)에 저장되며, OS에 의해 rest 상태에서 암호화된다.
  • Spending key는 트랜잭션 서명이 필요할 때만 seed에서 즉석에서 유도되고, FFI를 통해 Rust에 전달되며, 사용 후 secrecy::SecretVec으로 메모리 제로화된다.
  • mnemonic_to_seed()는 SecretVec<u8>을 반환한다. plain Vec<u8>이 아니므로 drop 시 자동 제로화된다.
  • 모든 API 함수 (api/wallet.rs, api/sync.rs)에 catch_unwind panic guard가 적용되어 있다. Rust panic이 발생해도 Flutter 프로세스가 크래시하지 않고 Result::Err("Rust panic: ...")으로 변환된다.

4. Dart / Flutter 레이어

4.1 Riverpod Providers

walletProvider (AsyncNotifierProvider<WalletNotifier, WalletState>)

  • build()에서: 월렛 DB 존재 여부를 확인하고, UFVK에서 주소를 로드
  • createWallet(): 니모닉을 생성하고, 계정을 만들고, 니모닉을 secure storage에 저장
  • importWallet(): 니모닉을 검증하고, 기존 DB를 삭제하고, 계정을 생성
  • WalletState: { hasWallet, unifiedAddress, network }

syncProvider (AsyncNotifierProvider<SyncNotifier, SyncState>)

  • startSync(): rust_sync.startFullSync() 호출 (blocking). 별도 Timer로 2초마다 진행률 폴링.
  • stopSync(): rust_sync.cancelFullSync() 호출 (AtomicBool 설정)
  • enableBackgroundSync(): iOS 26+ 전용. MethodChannel로 BGContinuedProcessingTask 제출.
  • SyncState: { isSyncing, isBackgroundMode, percentage, scannedHeight, chainTipHeight, transparentBalance, saplingBalance, orchardBalance, totalBalance, error }

4.2 Sync 실행

Sync 루프는 Rust에서 실행된다 (sync_engine.rs). Dart는 한 줄로 sync를 시작하고, 진행률을 폴링한다.

// sync_provider.dart (간소화)
await rust_sync.startFullSync(    // blocking — Rust에서 전체 sync 실행
  dbPath: dbPath,
  lightwalletdUrl: 'https://zec.rocks:443',
  network: 'main',
);

// 별도 Timer로 2초마다 폴링:
final progress = await rust_sync.getSyncStatus(dbPath: dbPath, network: 'main');
final balance = await rust_sync.getBalance(dbPath: dbPath, network: 'main');

Rust 내부 sync 루프 (run_sync_inner):

1. tonic gRPC로 zec.rocks:443에 TLS 연결 (tls-ring)
2. getLatestBlock() → update_chain_tip()
3. getSubtreeRoots() → put_subtree_roots() (incremental, start_index 최적화)
4. 루프:
   a. suggest_scan_ranges() → 최고 우선순위 범위 선택
   b. getBlockRange() → CompactBlock을 메모리에 저장 (MemoryBlockSource)
   c. getTreeState(start - 1) → ChainState 생성
   d. scan_cached_blocks(MemoryBlockSource) → trial decryption
   e. run_enhancement() → full tx 조회 + decrypt_and_store_transaction
   f. progress_fn 콜백 (C FFI용, Dart에서는 미사용)
5. 모든 범위 완료 → 반환

4.6 Enhancement 루프

매 scan batch 후 Rust sync 루프 내에서 실행된다. compact block에서 발견된 노트의 전체 TX 데이터를 lightwalletd에서 가져온다:

1. DB: transaction_data_requests() → 3가지 유형의 요청
2. 각 요청에 대해 (Rust에서 tonic gRPC 직접 호출):
   - Enhancement(txid): get_transaction(txid) → decrypt_and_store_transaction()
   - GetStatus(txid): get_transaction(txid) → set_transaction_status()
   - TransactionsInvolvingAddress(addr): get_taddress_txids(addr, range)
     → 각 TX에 대해 decrypt_and_store_transaction()
3. 최대 3라운드 반복, 더 이상 actionable 요청이 없으면 중단

Enhancement가 필요한 이유: compact block의 ciphertext는 앞 52바이트만 포함하므로 "내 노트인지" + "금액"은 알 수 있지만 메모 전체 내용은 알 수 없다. 풀 TX를 가져와야 메모, 정확한 금액, 발신자 정보 등을 얻을 수 있다.

TransactionsInvolvingAddress: transparent-inputs feature가 활성화되면 BIP-44 gap limit (기본 10개)까지의 transparent 주소에 대해 자동으로 TX 조회 요청이 생성된다. 이것은 lightwalletd 서버에 transparent 주소를 노출하므로 프라이버시 트레이드오프가 있다.

Non-actionable requests: TransactionsInvolvingAddress 요청 중 endHeight가 null인 것들은 전체 sync가 완료되지 않았기 때문이다. 이런 요청은 skip하고 다음 scan batch로 넘어가야 한다 (Android SDK도 동일하게 처리). 모든 요청이 non-actionable이면 enhancement 루프를 즉시 탈출한다.

Subtree roots 최적화: get_next_subtree_indices()로 이미 저장된 subtree root 개수를 확인하고, gRPC startIndex에 전달하여 새 root만 다운로드한다. 첫 sync에서는 ~1900개를 다운로드하지만, 이후 sync에서는 0~몇 개만 다운로드한다.

4.3 gRPC 클라이언트

gRPC 클라이언트는 Rust에서 tonic 크레이트로 실행된다. Dart에는 gRPC 코드가 없다.

zcash_client_backend의 lightwalletd-tonic feature가 CompactTxStreamerClient를 제공하며, sync_engine.rs에서 직접 사용한다. TLS는 tonic의 tls-ring + tls-webpki-roots feature로 처리된다 (rustls 0.23+ 요구사항).

사용하는 주요 RPC 메서드 (모두 Rust에서 호출):

  • get_latest_block() — 현재 chain tip
  • get_block_range() — compact block 스트리밍 다운로드
  • get_tree_state() — 특정 높이의 commitment tree 상태
  • get_subtree_roots() — subtree root 스트리밍 다운로드
  • get_transaction() — 특정 TX의 풀 데이터 조회 (enhancement)
  • get_taddress_txids() — transparent 주소 관련 TX 목록 조회

5.2 Compact Block이란

Compact Block은 Zcash 블록의 경량화 버전이다. 라이트 클라이언트가 trial decryption을 수행하는 데 필요한 최소한의 데이터만 포함한다.

Full BlockCompact Block
크기~2MB (평균)~2KB (평균)
포함 내용모든 TX 데이터, 서명, 스크립트, 풀 메모블록 헤더 + shielded 출력의 암호화된 부분만
용도풀노드 검증라이트 클라이언트 trial decryption
대역폭100%~1% (90%+ 절약)

왜 필요한가: Shielded 트랜잭션에서 "이것이 내 트랜잭션인지" 확인하려면 모든 shielded 출력을 내 viewing key로 복호화 시도해야 한다 (trial decryption). 이 작업에 필요한 데이터는 각 출력의 암호화된 ciphertext와 ephemeral key 뿐이다. 서명, transparent 스크립트, 풀 메모 등은 불필요하다.

lightwalletd가 풀노드에서 블록을 받아 불필요한 부분을 걷어내고 compact block으로 변환하여 라이트 클라이언트에 제공한다.

Protobuf 구조:

message CompactBlock {
  uint64 height = 2;
  bytes  hash = 3;            // 블록 해시
  bytes  prevHash = 4;        // 이전 블록 해시 (체인 검증용)
  uint32 time = 5;
  repeated CompactTx vtx = 7; // 트랜잭션 목록
}

message CompactTx {
  bytes hash = 1;                             // TX 해시
  repeated CompactSaplingOutput outputs = 5;  // Sapling 출력
  repeated CompactOrchardAction actions = 6;  // Orchard 액션
}

message CompactSaplingOutput {
  bytes cmu = 1;           // note commitment
  bytes ephemeralKey = 2;  // 복호화에 필요
  bytes ciphertext = 3;    // 암호화된 노트 (앞 52바이트만)
}

ciphertext의 한계: compact block의 ciphertext는 풀 메모(512바이트)가 아니라 앞 52바이트만 포함한다. 이것으로 "내 노트인지" + "금액이 얼마인지"는 알 수 있지만, 메모 전체 내용은 알 수 없다. 메모를 읽으려면 enhancement 루프에서 풀 트랜잭션을 별도로 가져와야 한다.

5.3 In-Memory Block Source

이전에는 FsBlockDb(파일시스템 기반 캐시)를 사용했으나, 현재는 메모리 기반 MemoryBlockSource를 사용한다.

zcash_client_backend의 BlockSource trait을 구현하여, gRPC에서 받은 CompactBlock을 Vec에 저장하고 scan_cached_blocks()에 직접 전달한다. 파일 I/O가 완전히 제거되었다.

// sync_engine.rs
struct MemoryBlockSource {
    blocks: Vec<CompactBlock>,  // gRPC에서 받은 블록
}

impl BlockSource for MemoryBlockSource {
    fn with_blocks(..., mut with_block: F) {
        for block in &self.blocks {
            with_block(block.clone())?;
        }
    }
}

이로 인해 cache_path 파라미터, FsBlockDb, BlockMeta, init_blockmeta_db가 모두 불필요해졌다.

5.4 Trial Decryption & 노트 발견

zcash_client_backend의 scan_cached_blocks()가 캐시된 블록의 모든 shielded 출력에 trial decryption을 수행한다:

  • 각 Sapling 출력에 대해: 월렛의 Sapling IVK로 복호화 시도
  • 각 Orchard 액션에 대해: 월렛의 Orchard IVK로 복호화 시도
  • 복호화 성공 = 이 월렛의 노트 → 월렛 DB에 저장
  • Nullifier를 추적하여 노트가 사용되었는지 감지

5.5 진행률 보고

Dart가 2초 간격 Timer로 get_sync_status()와 get_balance()를 폴링한다. Rust sync 루프 내에서도 매 batch마다 progress_fn 콜백이 호출되지만, Dart FRB 경로에서는 이 콜백을 사용하지 않는다 (C FFI 경로에서 iOS task.progress 업데이트에 사용).

7. 주소 체계

7.1 Unified Address (Sapling + Orchard)

사용자에게 표시되는 기본 주소는 Sapling과 Orchard receiver만 포함하는 ZIP-316 Unified Address이다 (transparent 제외). zodl/Zashi 동작과 일치한다.

  • 메인넷 접두사: u1...
  • Receivers: Orchard (Pallas 곡선) + Sapling (Jubjub 곡선)
  • 보내는 월렛이 자동으로 지원하는 가장 높은 프라이버시 레벨의 receiver를 선택

7.2 Transparent Address

getTransparentAddress()를 통해 별도의 transparent 전용 주소를 조회할 수 있다. 이것은 동일한 seed에서 BIP-44 경로 m/44'/133'/0'로 유도된 표준 t-address (P2PKH, 메인넷 접두사 t1)이다.

7.3 zodl/Zashi 호환성

같은 니모닉으로 생성해도 우리 월렛과 zodl에서 다른 주소가 생성된다. 이것은 예상된 동작이다 — zodl의 Android SDK가 [patch.crates-io]를 통해 crates.io 공개 버전과 다른 특정 git revision의 orchard와 sapling-crypto를 사용하기 때문이다. 키 유도 수학이 다른 곡선 포인트를 생성한다. 두 주소 모두 동일한 seed에서 유도된 유효한 Zcash 주소이다.

9. Background Sync

Background sync는 iOS 26+ 전용이다. BGContinuedProcessingTask를 사용하며, Swift가 C FFI로 Rust sync를 직접 실행한다.

9.1 iOS BGContinuedProcessingTask (iOS 26+)

핵심 원리: Dart 코드는 iOS background에서 실행할 수 없다. BGContinuedProcessingTask의 핸들러는 Swift에서 실행되므로, Rust를 C FFI로 직접 호출해야 한다.

Dart: "Sync on Background" 버튼 탭 → MethodChannel('com.zcash.wallet/background_sync') → Swift: BGContinuedProcessingTaskRequest 제출 → OS가 핸들러 실행 (background queue) → zcash_run_full_sync() [C FFI, blocking] → tokio runtime 생성 → run_sync_inner() [gRPC + scan + enhancement] → C callback → task.progress 업데이트 (NSProgress, thread-safe) → C callback → SyncProgressStreamHandler → EventChannel → Dart (앱이 foreground면) → task.setTaskCompleted(success:)

주요 파일:

파일역할
rust/src/ffi.rsC FFI: zcash_run_full_sync(), zcash_cancel_sync()
ios/Runner/zcash_sync.hC 헤더 (Swift Bridging Header에서 import)
ios/Runner/BackgroundSyncManager.swift@available(iOS 26.0, *). BGTask 등록/제출/핸들링
ios/Runner/AppDelegate.swifttask 등록 + MethodChannel + EventChannel 설정
ios/Runner/SyncProgressStreamHandler.swiftC callback → FlutterEventChannel 브릿지

task.progress (필수): BGContinuedProcessingTask는 NSProgressReporting을 준수하며, progress 보고가 필수이다. stalled로 판단되면 시스템이 task를 강제 종료한다. C callback에서 BackgroundSyncManager.shared.taskProgress (NSProgress 참조)를 thread-safe하게 업데이트한다.

시스템 UI: OS가 자동으로 Dynamic Island / Lock Screen에 title, subtitle, progress를 표시한다. 별도의 Live Activity 구현이 불필요하다.

expirationHandler: 시스템이 시간 만료 또는 사용자가 Stop을 누르면 호출. zcash_cancel_sync()로 AtomicBool을 설정하여 Rust sync 루프가 다음 batch에서 정상 종료한다.

필수 설정:

  • Info.plist: UIBackgroundModes (processing, fetch) + BGTaskSchedulerPermittedIdentifiers (com.zcash.zcashWallet.sync)
  • Runner-Bridging-Header.h: #import "zcash_sync.h"
  • Android: background sync 미지원 (foreground sync만 가능)

9.2 Android

Android에서는 background sync를 지원하지 않는다. Foreground에서만 sync가 실행된다.

10. 알려진 주의사항

주제주의사항
TxId byte order Rust에서 TxId::as_ref()가 원본 바이트를 반환하므로 reverse 불필요. TxId::Display는 byte-reversed hex (블록 탐색기 형식).
orchard/sapling 버전 zcash_client_sqlite가 사용하는 버전과 반드시 일치해야 함 (현재 orchard 0.11, sapling-crypto 0.5). 다르면 Rust 타입 불일치 에러.
Sapling 활성화 높이 419,200 이전 블록에서 getTreeState() 호출 시 lightwalletd가 에러 반환. 빈 hash로 ChainState::empty() 사용.
rustls CryptoProvider tonic의 tls-webpki-roots는 crypto backend를 포함하지 않음. tls-ring feature를 반드시 함께 활성화해야 rustls 0.23+에서 동작.
C callback context C function pointer는 context를 capture할 수 없다. iOS background sync에서 task.progress를 업데이트하려면 NSProgress 참조를 shared 변수에 저장하고 C callback에서 접근해야 한다.
ffi.rs 위치 ffi.rs는 api/ 밖에 위치해야 함. FRB codegen이 api/ 하위의 extern "C" 함수를 처리하려다 실패.

11. 개발 명령어

# Flutter (항상 fvm 사용)
fvm flutter run                  # 연결된 디바이스/시뮬레이터에서 실행
fvm flutter test                 # Dart 단위 테스트
fvm flutter analyze              # 정적 분석

# Rust
cd rust && cargo test            # Rust 단위 테스트
cd rust && cargo check           # 빠른 컴파일 체크

# FRB codegen (rust/src/api/*.rs 변경 후)
flutter_rust_bridge_codegen generate   # 프로젝트 루트에서 실행!

# iOS 시뮬레이터
./clear-app.sh                   # 시뮬레이터에서 앱 + 키체인 초기화

4. Dart / Flutter 레이어

4.1 Riverpod Providers

accountProvider (AsyncNotifierProvider<AccountNotifier, AccountState>)

멀티 계정의 중심 provider. 계정 목록, 활성 계정, per-account 니모닉을 관리한다.

  • build()에서: secure storage에서 계정 목록(zcash_accounts)과 활성 계정(zcash_active_account)을 로드. 활성 계정의 주소를 Rust에서 조회.
  • createAccount({name?}): 첫 번째 계정이면 create_wallet() (Derived), 추가 계정이면 generate_mnemonic() + add_account() (Imported). birthday는 chain tip에서 가져옴. 니모닉은 zcash_account_mnemonic_{uuid}로 per-account 저장.
  • importAccount({mnemonic, birthdayHeight?, name?}): 첫 번째이면 import_wallet(), 추가이면 add_account().
  • importKeystoneAccount({name, ufvk, seedFingerprint, zip32Index}): 하드웨어 UFVK 만 으로 계정 추가. 빈 계정 목록에서는 거부 (하드웨어-first 부트스트랩 금지 — §9.3). isHardware: true 로 마킹. zcash_account_mnemonic_{uuid} 엔트리는 만들지 않는다.
  • switchAccount(uuid): 활성 계정 전환. secure storage 업데이트 + 주소 조회.
  • renameAccount(uuid, newName): 계정 이름 변경. AccountInfo.copyWith 로 필드 보존 (특히 isHardware — 직접 생성자 호출은 field-drop 버그 주의).
  • getActiveMnemonic(): 활성 계정의 니모닉을 secure storage에서 조회. 전송 시 seed 유도에 사용. 하드웨어 계정에 대해서는 null 반환.
  • isActiveAccountHardware: 활성 계정의 isHardware 플래그. Send screen 이 이 값으로 execute_proposal (software) vs PCZT 파이프라인 (hardware) 경로를 분기.
  • AccountState: { accounts: List<AccountInfo>, activeAccountUuid, activeAddress }. AccountInfo 는 { uuid, name, order, isHardware } 를 가지며 copyWith(name?, order?) 를 제공한다.

Secure Storage 키 구조:

  • zcash_accounts — 계정 목록 JSON ([{uuid, name, order}, ...])
  • zcash_account_mnemonic_{uuid} — 계정별 니모닉
  • zcash_active_account — 현재 활성 계정 UUID
  • zcash_wallet_network — 네트워크 (main/test)

walletProvider (AsyncNotifierProvider<WalletNotifier, WalletState>)

  • build()에서: ref.watch(accountProvider)로 AccountProvider의 상태를 구독하여 자동 갱신. 계정이 있으면 hasWallet=true.
  • WalletState: { hasWallet, unifiedAddress, network, activeAccountUuid }
  • 직접 계정 생성/가져오기 로직은 없음 — 모두 AccountProvider에 위임.

syncProvider (AsyncNotifierProvider<SyncNotifier, SyncState>)

자동 Sync:

  • build()에서 ref.listen(accountProvider)로 계정 수 변화 감지 + ref.read로 초기 체크 — rebuild 없이 startSync() + _startPolling()
  • startSync()은 fire-and-forget: FRB stream listener를 등록하고 즉시 return. Completer 없음
  • _syncGen generation counter: stopSync()가 증가시키고, .then() 콜백에서 체크하여 유저 stop 후 pending 작업 무효화
  • _checkAndSync(): 10초마다 getLatestBlockHeight() polling → 새 블록 감지 또는 이전 sync 미완료 시 재sync
  • 중복 방지: _isSyncing (Dart) + isSyncRunning() (Rust) 이중 가드

Sync 제어:

  • stopSync(): ++_syncGen + cancelFullSync() + _stopPolling(). 다음 app resume에서 재개
  • Background sync는 BackgroundSyncDelegate에 위임 (Platform 분기 없음)
  • disableBackgroundSync(): delegate.disable()가 bool 반환 — iOS=true(fg 재시작 필요), Android=false

BackgroundSyncDelegate (background_sync_delegate.dart):

  • Abstract 인터페이스 + Android/iOS/NoOp 구현체. 모든 Platform.isAndroid/Platform.isIOS 분기가 여기에 격리됨
  • shouldSuppressPolling: Android=false (notification뿐), iOS=_active (BGTask가 sync 관리)
  • Android onSyncDone(): _active 유지 (notification은 유저가 disable할 때까지 지속)
  • iOS onResume(): bg sync 완료/expiration 감지 → _active 리셋

Lifecycle:

  • onResume: balance 새로고침 → delegate.onResume() → _checkAndSync() (내부에서 polling 시작)
  • onHide: polling 중단
  • Polling 시작 경로는 3곳만: _onSyncDone, _checkAndSync, 계정 생성 시

4.2 Sync 실행

Sync 루프는 Rust에서 실행된다 (sync_engine.rs). Dart는 startFullSync(mode: 1)을 호출하여 Stream<ApiSyncProgressEvent>를 수신하고, stream.listen()으로 진행률 콜백을 등록한다.

// sync_provider.dart (간소화)
// startSync()은 fire-and-forget — stream listener를 등록하고 즉시 return
void startSync() {
  if (_isSyncing || rust_sync.isSyncRunning()) return;
  _isSyncing = true;
  final gen = ++_syncGen;  // stopSync가 증가시켜서 pending 콜백 무효화

  _getDbPath().then((dbPath) {
    if (gen != _syncGen) return; // stopSync 호출됨
    final stream = rust_sync.startFullSync(...);
    _syncSub = stream.listen(
      _onSyncProgress,
      onDone: () {
        if (gen != _syncGen) return;
        _isSyncing = false;
        _onSyncDone();  // → _refreshBalance() + _startPolling()
      },
      onError: (e) {
        if (gen != _syncGen) return;
        _isSyncing = false;
        state = AsyncData(SyncState(error: e.toString()));
        _startPolling(); // 에러 후에도 polling 유지 → 자동 재시도
      },
    );
  });
}
// Sync는 모든 계정을 스캔하지만, 잔액/내역은 activeAccountUuid로 필터
// Polling 10초마다 getLatestBlockHeight → 새 블록 또는 미완료 시 재sync

Rust 내부 sync (run_sync_inner → run_sync_impl):

run_sync_inner (retry wrapper):
  SYNC_START 설정 (한 번만, retry 간 elapsed 일관)
  최대 4회 시도 (1 + 3 retry, backoff 2s/4s/8s)
  retry 대기 중에도 cancel/mode 체크
  모든 시도 실패 → 마지막 에러 반환 → Dart에서 sync.error로 UI 표시

run_sync_impl (단일 시도):
  1. tonic gRPC로 zec.rocks:443에 TLS 연결 (tls-ring)
  2. getLatestBlock() → update_chain_tip()
  3. getSubtreeRoots() → put_subtree_roots() (incremental)
  4. suggest_scan_ranges()로 initial_total 계산 (스캔 전 전체 블록 수)
  5. 루프 (100블록 batch):
     a. cancel/mode 체크
     b. suggest_scan_ranges() → 최고 우선순위 범위 선택
     c. getBlockRange() → MemoryBlockSource
     d. cancel/mode 체크
     e. getTreeState(start - 1) → ChainState
     f. scan_cached_blocks(MemoryBlockSource)
     g. cancel/mode 체크
     h. run_enhancement()
     i. remaining 계산 → pct = 1.0 - remaining / initial_total
     j. has_new_tx 판별 (ScanSummary note count > 0)
     k. progress_fn 콜백 → StreamSink (Dart) 또는 C callback (Swift)
  6. 모든 범위 완료 → final progress (pct=1.0, is_complete=true)

4.6 Enhancement 루프

매 scan batch 후 Rust sync 루프 내에서 실행된다. compact block에서 발견된 노트의 전체 TX 데이터를 lightwalletd에서 가져온다:

1. DB: transaction_data_requests() → 3가지 유형의 요청
2. 각 요청에 대해 (Rust에서 tonic gRPC 직접 호출):
   - Enhancement(txid): get_transaction(txid) → decrypt_and_store_transaction()
   - GetStatus(txid): get_transaction(txid) → set_transaction_status()
   - TransactionsInvolvingAddress(addr): get_taddress_txids(addr, range)
     → 각 TX에 대해 decrypt_and_store_transaction()
3. 최대 3라운드 반복, 더 이상 actionable 요청이 없으면 중단

Enhancement가 필요한 이유: compact block의 ciphertext는 앞 52바이트만 포함하므로 "내 노트인지" + "금액"은 알 수 있지만 메모 전체 내용은 알 수 없다. 풀 TX를 가져와야 메모, 정확한 금액, 발신자 정보 등을 얻을 수 있다.

TransactionsInvolvingAddress: transparent-inputs feature가 활성화되면 BIP-44 gap limit (기본 10개)까지의 transparent 주소에 대해 자동으로 TX 조회 요청이 생성된다. 이것은 lightwalletd 서버에 transparent 주소를 노출하므로 프라이버시 트레이드오프가 있다.

Non-actionable requests: TransactionsInvolvingAddress 요청 중 endHeight가 null인 것들은 전체 sync가 완료되지 않았기 때문이다. 이런 요청은 skip하고 다음 scan batch로 넘어가야 한다 (Android SDK도 동일하게 처리). 모든 요청이 non-actionable이면 enhancement 루프를 즉시 탈출한다.

Subtree roots 최적화: get_next_subtree_indices()로 이미 저장된 subtree root 개수를 확인하고, gRPC startIndex에 전달하여 새 root만 다운로드한다. 첫 sync에서는 ~1900개를 다운로드하지만, 이후 sync에서는 0~몇 개만 다운로드한다.

4.3 gRPC 클라이언트

gRPC 클라이언트는 Rust에서 tonic 크레이트로 실행된다. Dart에는 gRPC 코드가 없다.

zcash_client_backend의 lightwalletd-tonic feature가 CompactTxStreamerClient를 제공하며, sync_engine.rs에서 직접 사용한다. TLS는 tonic의 tls-ring + tls-webpki-roots feature로 처리된다 (rustls 0.23+ 요구사항).

사용하는 주요 RPC 메서드 (모두 Rust에서 호출):

  • get_latest_block() — 현재 chain tip
  • get_block_range() — compact block 스트리밍 다운로드
  • get_tree_state() — 특정 높이의 commitment tree 상태
  • get_subtree_roots() — subtree root 스트리밍 다운로드
  • get_transaction() — 특정 TX의 풀 데이터 조회 (enhancement)
  • get_taddress_txids() — transparent 주소 관련 TX 목록 조회

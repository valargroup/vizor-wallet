9. Background Sync

Background sync는 iOS와 Android 모두 지원한다. iOS는 BGContinuedProcessingTask + C FFI로 Rust sync를 직접 실행하고, Android는 flutter_foreground_task를 통해 notification만 관리하며 sync 자체는 동일한 FRB Stream으로 실행된다.

9.1 iOS BGContinuedProcessingTask (iOS 26+)

핵심 원리: Dart 코드는 iOS background에서 실행할 수 없다. BGContinuedProcessingTask의 핸들러는 Swift에서 실행되므로, Rust를 C FFI로 직접 호출해야 한다.

Mode 전환 메커니즘:

Rust에 DESIRED_SYNC_MODE AtomicU8이 존재한다 (0=none, 1=foreground, 2=background). 이 값으로 foreground/background sync 간 전환을 제어한다.

  • Foreground → Background: Dart가 setSyncMode(2) 호출 → Rust fg sync가 다음 batch에서 mode mismatch 감지 후 종료 → Swift BGTask handler가 while(mode!=2 || is_running) sleep(2s)로 대기 → 조건 충족 시 C FFI zcash_run_full_sync() 호출로 bg sync 시작
  • Background → Foreground: Dart가 setSyncMode(1) 호출 → Rust bg sync가 다음 batch에서 mode mismatch 감지 후 종료 → Dart가 while(isSyncRunning()) await 200ms로 대기 → fg sync 시작
  • Expiration: expirationHandler가 heartbeat 타이머 취소, mode=0 설정, sync cancel → resubmit 하지 않음 → 앱이 foreground로 돌아오면 AppLifecycleListener.onResume에서 mode==0 감지 → foreground sync 재시작
  • BGTask resubmit: sync가 종료되었는데 mode가 여전히 2이면 BGTask를 다시 제출하여 background sync 계속

BGTask 등록 및 핸들러 구조

BGTask 등록: using: nil로 등록한다 (syncQueue가 아님). 이렇게 하면 expirationHandler가 다른 queue에서 실행될 수 있어 deadlock을 방지한다.

핸들러 실행 흐름:

  • Handler는 semaphore 패턴으로 .utility QoS syncQueue에 작업을 dispatch한다
  • C FFI는 current_thread tokio runtime을 사용 — dispatch queue의 .utility QoS를 상속한다
  • Wait loop에 120초 timeout이 있다

Heartbeat timer: .global(qos: .utility)에서 실행되며, 5초마다 completedUnitCount를 +1 nudge한다. 이는 OS가 task를 stalled로 판단하지 않도록 보장한다.

Batch callback: completedUnitCount = percentage * 10000으로 설정한다 (scan-queue 기반, 0-10000 scale). EventChannel을 통해 scannedHeight, chainTipHeight, percentage, isSyncing, isComplete, hasNewTx를 Dart에 전달한다.

expirationHandler: heartbeat 타이머 취소, mode=0 설정, sync cancel을 수행한다.

C FFI 동작: C FFI는 DESIRED_SYNC_MODE를 강제로 설정하지 않는다. 시작 전에 mode==2인지 확인하고, 아니면 error code 4를 반환한다. 모든 error path에서 로깅이 수행된다.

Dart: "Sync on Background" 버튼 탭 → setSyncMode(2) [FRB] → Rust fg sync: mode mismatch → 종료 → MethodChannel → Swift: BGContinuedProcessingTaskRequest 제출 → OS가 핸들러 실행 (using: nil — handler thread) → semaphore 패턴으로 .utility QoS syncQueue에 dispatch → while(mode!=2 || is_running) sleep(2s) [대기, 120s timeout] → zcash_run_full_sync() [C FFI, blocking] → current_thread tokio runtime (inherits .utility QoS) → run_sync_inner() [gRPC + scan + enhancement] → C callback → completedUnitCount = percentage * 10000 (scan-queue based) → C callback → SyncProgressStreamHandler → EventChannel → Dart (앱이 foreground면) (isSyncing, isComplete, hasNewTx 포함) → heartbeat timer on .global(.utility): +1 nudge every 5s → sync 종료, mode==2이면 BGTask resubmit → task.setTaskCompleted(success:)

9.2 TX Tracking BGTask (iOS 26+)

전송 후 트랜잭션이 채굴되었는지 추적하기 위한 별도의 BGContinuedProcessingTask이다. Sync BGTask와는 독립적으로 동작한다.

핵심 동작:

  • TxTrackManager가 com.zcash.zcashWallet.txtrack BGTask를 등록/관리
  • 전송 성공 후 BGTask를 제출하면, background에서 lightwalletd를 주기적으로 폴링하여 TX 상태 확인
  • TX가 채굴되면 (mined) 또는 만료되면 (expired) 추적 종료

DynamicIslandManager

DynamicIslandManager는 Live Activity / Dynamic Island의 표시 우선순위를 관리한다. Sync progress와 TX tracking이 동시에 활성화될 수 있으므로, 어느 것을 Dynamic Island에 표시할지 결정해야 한다.

  • DisplayMode: idle, sync, txTrack 세 가지 상태
  • TX tracking이 sync보다 우선: displayMode == .txTrack이면 sync progress가 Live Activity를 업데이트하지 않음 (UserDefaults는 항상 업데이트)
  • TX tracking 완료 후 마지막 sync progress로 복원 (lastSyncPercentage 캐시)
  • UserDefaults (group.com.zcash.zcashWallet)를 통해 Widget과 데이터 공유

9.3 Android Foreground Service

Android에서는 flutter_foreground_task를 사용하여 notification만 관리한다. Sync 자체는 foreground와 동일한 Dart FRB Stream으로 실행되며, mode 전환 메커니즘은 없다.

  • enableBackgroundSync(): foreground service notification을 추가한다. 사용자에게 sync가 진행 중임을 알리고 시스템이 앱을 kill하지 않도록 한다.
  • disableBackgroundSync(): notification을 제거한다. Sync 자체는 중단되지 않는다.

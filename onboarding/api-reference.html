6.3 Send 함수

전송은 software 와 hardware 두 경로가 있다. 공통 첫 단계는 estimate_fee / propose_send. 이후 AccountProvider.isActiveAccountHardware 로 분기한다. 모든 함수가 account_uuid / proposal_id 를 받아 계정별로 동작한다. ZEC 금액은 integer-only 파싱으로 zatoshi로 변환 (부동소수점 사용하지 않음).

Hardware 경로 (Keystone PCZT 파이프라인)

자세한 흐름과 role 분담은 §9.4 Three-PCZT 파이프라인 참조.

Dart FunctionRust Function설명
createPcztFromProposal({dbPath, network, proposalId}) create_pczt_from_proposal(...) Proposal 을 base PCZT (Creator + IoFinalizer 역할) 로 변환. PROPOSAL_STORE 에서 consume-on-entry (execute_proposal 과 동일 패턴). Bytes 반환.

Combiner → TransactionExtractor (verifying key 사용) → gRPC broadcast → primary storage extract_and_store_transaction_from_pczt, 실패 시 fallback decrypt_and_store_transaction. txid 반환. 반드시 broadcast 뒤에 store. Sapling params 는 addProofsToPczt 와 동일한 쌍 으로 전달 — 한쪽만 넘기면 SaplingRequired 로 실패 (§9.6 invariants).

Sapling Proving Parameters: Sapling proof 생성에 sapling-spend.params (~47MB) + sapling-output.params (~3.5MB) 파일이 필요하다. Orchard (Halo 2)는 trusted setup이 없어 파라미터 파일이 불필요하다. propose_send()의 needsSaplingParams 플래그로 필요 여부를 사전에 확인한 후, 필요 시 https://download.z.cash/downloads/에서 다운로드하고 SHA-1 검증 후 저장한다. 한 번만 다운로드하면 이후에는 로컬 파일을 사용한다.

Software vs hardware: Software 경로의 executeProposal 은 needsSaplingParams=false 일 때 no-op Sapling prover 를 써서 params 파일이 필요 없다. Hardware 경로는 needsSaplingParams=true 일 때 같은 params 경로를 addProofsToPczt 와 extractAndBroadcastPczt 양쪽에 넘겨야 한다 (proving key + verifying key 둘 다 필요).

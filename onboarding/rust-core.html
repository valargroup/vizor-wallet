api/ 레이어가 FFI 경계이며, 파라미터를 파싱하고 wallet/에 위임하는 얇은 래퍼만 포함한다. wallet/ 레이어에 실제 Zcash 로직이 들어있다.

두 가지 FFI 경로: Dart는 FRB를 통해 api/sync.rs의 start_full_sync()를 호출한다. iOS background에서는 Swift가 C FFI를 통해 ffi.rs의 zcash_run_full_sync()를 호출한다. 둘 다 내부적으로 sync_engine::run_sync_inner()를 실행한다.

ffi.rs가 api/ 밖에 있는 이유: FRB codegen이 api/ 하위의 모든 pub fn을 Dart 바인딩으로 변환하는데, C FFI 함수(extern "C", *const c_char 등)는 FRB가 처리할 수 없다.

3.2 Wallet API — 키 유도 & 계정 관리

파일: rust/src/wallet/keys.rs

니모닉 생성: bip0039 크레이트로 256비트 엔트로피의 24단어 BIP-39 니모닉을 생성한다. Seed 유도는 표준을 따른다: PBKDF2-HMAC-SHA512, salt = "mnemonic" + passphrase (빈 passphrase).

키 유도: zcash_keys::UnifiedSpendingKey::from_seed()가 하나의 seed에서 세 종류의 키를 유도한다:

  • Transparent: BIP-44 경로 m/44'/133'/0' (mainnet)
  • Sapling: Jubjub 곡선 위의 ZIP-32 HD 유도
  • Orchard: Pallas 곡선 위의 ZIP-32 HD 유도

계정 생성 (멀티 계정 모델): 단일 DB에 여러 계정이 공존한다. 두 가지 계정 유형이 있다:

  • Derived (첫 번째 계정): WalletDb::create_account()으로 생성. seed relevance check를 위해 최소 하나의 Derived 계정이 필요하다.
  • Imported (추가 계정): WalletDb::import_account_ufvk()로 생성. 다른 seed에서 USK를 유도하고 UFVK를 추출하여 AccountPurpose::Spending { derivation: Some(Zip32Derivation) }로 import한다. 이렇게 하면 서로 다른 seed의 계정이 같은 DB에 공존할 수 있다 (create_account은 단일 seed만 허용).

DB에는 UFVK만 저장되며, spending key는 영속화하지 않는다. 각 계정은 AccountUuid로 식별된다.

주소 인코딩 (software 계정): Unified Address는 Require Orchard + Require Sapling + Omit Transparent로 생성하며, zodl/Zashi 월렛 동작과 일치한다. shielded_address_request() 헬퍼 함수가 이를 제어한다.

fn shielded_address_request() -> UnifiedAddressRequest {
    UnifiedAddressRequest::custom(
        ReceiverRequirement::Require, // Orchard
        ReceiverRequirement::Require, // Sapling
        ReceiverRequirement::Omit,    // Transparent
    ).expect("valid receiver requirements")
}

주소 인코딩 (Keystone 계정): Keystone 디바이스가 UFVK 의 Sapling 부분을 export 하지 않으므로, shielded_address_request 를 그대로 쓰면 "Sapling receiver 가 필요한데 없음" 으로 실패한다. import_hardware_account 는 orchard_address_request 로 fallback 한다 (Sapling 을 Omit 으로). 결과 UA 는 Orchard receiver 하나만 포함. 자세한 것은 Keystone §9.2 참고.

fn orchard_address_request() -> UnifiedAddressRequest {
    UnifiedAddressRequest::custom(
        ReceiverRequirement::Require, // Orchard
        ReceiverRequirement::Omit,    // Sapling — Keystone UFVK 에 없음
        ReceiverRequirement::Omit,    // Transparent
    ).expect("valid receiver requirements")
}

3.3 Sync API — 블록 스캐닝 & 잔액

파일: rust/src/wallet/sync.rs, rust/src/wallet/sync_engine.rs

Sync는 Rust에서 통합 실행된다. sync_engine.rs의 run_sync_inner()가 gRPC 연결, 블록 다운로드, 스캔, enhancement를 모두 처리한다. Dart는 startFullSync(mode: 1)을 호출하면 Stream<ApiSyncProgressEvent>를 반환받아 stream.listen()으로 진행률을 수신한다. 폴링은 없으며, 모든 상태 업데이트는 stream, EventChannel, onResume으로 구동된다.

Function역할
start_full_sync(mode)통합 sync 실행 (gRPC + scan + enhancement). Stream<ApiSyncProgressEvent> 반환. mode 파라미터로 foreground(1)/background(2) 구분.
cancel_full_sync()AtomicBool로 sync 취소 신호. sync
get_balance(account_uuid)계정별 WalletBalance { transparent, sapling, orchard, transparent_pending, sapling_pending, orchard_pending, total } 반환 (zatoshi). total은 confirmed + pending 합산.
get_transaction_history(account_uuid, limit)계정별 v_transactions SQL view에서 TX 내역 조회. limit 파라미터로 최근 N건 제한 가능.
validate_address()주소 타입 검증 (unified/sapling/transparent/invalid)
get_next_available_address(account_uuid)계정별 다음 diversified UA 생성
propose_send(account_uuid)계정별 전송 제안: 노트 선택 + ZIP-317 수수료 계산. Integer-only ZEC-to-zatoshi 변환 (부동소수점 사용하지 않음). StoredProposal에 account_id 포함. PROPOSAL_STORE 에 insert.
estimate_fee(account_uuid)계정별 수수료 추정 (proposal 저장 없이). 전송 전 실제 수수료를 사용자에게 표시.
execute_proposal()(software) 제안 실행: zk-SNARK proof 생성 + 서명. Orchard-only TX에는 no-op Sapling prover 사용. TX broadcast via gRPC. 친화적 에러 메시지 반환. PROPOSAL_STORE 에서 consume-on-entry.

redact_pczt_for_signer(pczt)(hardware) Witness / proprietary metadata 제거 → 하드웨어 디바이스에 넘길 minimized PCZT.

rewind_to_height()chain reorg 시 롤백

Legacy 함수: api/sync.rs에는 이전 Dart-orchestrated sync에서 사용하던 개별 단계 함수들(update_chain_tip, suggest_scan_ranges, write_block_metadata, scan_blocks 등)이 아직 남아있다. FRB 바인딩은 생성되지만 Dart에서 호출되지 않는 dead code이다. PCZT 파이프라인 함수들 (create_pczt_from_proposal, add_proofs_to_pczt, redact_pczt_for_signer, extract_and_broadcast_pczt, discard_proposal) 은 legacy 가 아니라 현재 사용 중인 하드웨어 송금 경로이다.

3.4 Zcash Crate 의존성 그래프

zcash_client_sqlite (0.20.2) <-- 우리의 주요 진입점 |-- zcash_client_backend (0.22.0) | |-- zcash_keys (0.13.0) | |-- zcash_protocol (0.8.0) | |-- zcash_address (0.11.0) | |-- zcash_primitives (0.27.1) | +-- lightwalletd proto (내부 tonic gRPC) |-- sapling-crypto (0.7.0) <-- zcash_client_sqlite의 버전과 반드시 일치해야 함 +-- orchard (0.13.1) <-- zcash_client_sqlite의 버전과 반드시 일치해야 함 tonic (0.14) <-- gRPC 클라이언트 (tls-ring + tls-webpki-roots)

버전 제약: orchard와 sapling-crypto 버전은 zcash_client_sqlite가 내부적으로 사용하는 것과 반드시 일치해야 한다. 다른 버전을 사용하면 타입 불일치 에러가 발생한다 (예: "expected sapling_crypto::Node, found Node"). Rust는 서로 다른 crate 버전을 다른 타입으로 취급하기 때문이다.

3.5 보안 모델: Spending Key 분리

librustzcash의 의도적인 설계이며, 우리의 선택이 아니다. WalletDb 에는 Unified Full Viewing Key (UFVK) 만 저장된다 — 트랜잭션 감지와 잔액 계산에는 충분하지만, 자금을 사용할 수는 없다.

소프트웨어 계정:

  • 니모닉 / seed는 Dart의 flutter_secure_storage(iOS Keychain / Android Keystore)에 계정별로 저장된다 (zcash_account_mnemonic_{uuid}). OS에 의해 rest 상태에서 암호화된다.
  • Spending key는 트랜잭션 서명이 필요할 때만 seed에서 즉석에서 유도되고, FFI를 통해 Rust에 전달되며, 사용 후 secrecy::SecretVec으로 메모리 제로화된다.
  • mnemonic_to_seed()는 SecretVec<u8>을 반환한다. plain Vec<u8>이 아니므로 drop 시 자동 제로화된다.

Keystone 하드웨어 계정:

  • Phone 에 seed / USK 가 절대 존재하지 않는다. Import 시 UFVK 만 QR/UR 로 받아서 DB 에 저장하고, 서명은 매 트랜잭션마다 디바이스로 PCZT 를 넘겨서 수행한다.
  • zcash_account_mnemonic_{uuid} 엔트리가 없고, AccountProvider.getActiveMnemonic() 은 하드웨어 계정에 대해 null 을 반환한다.
  • 보안 관점에서 "송금 순간에도 phone 이 USK 를 일시적으로라도 보유하지 않는다" 는 것이 하드웨어 지갑의 핵심 보장이다. 우리 구현은 이 보장을 지킨다 — 상세는 §9 Keystone 하드웨어 지갑.

모든 API 함수 (api/wallet.rs, api/sync.rs, api/keystone.rs) 에 catch_unwind panic guard 가 적용되어 있다. Rust panic 이 발생해도 Flutter 프로세스가 크래시하지 않고 Result::Err("Rust panic: ...") 으로 변환된다.

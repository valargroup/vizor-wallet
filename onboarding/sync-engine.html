5. Sync 엔진 상세

5.1 전체 Sync 흐름

메인넷 초기 sync는 Sapling 활성화 높이(419,200)부터 시작하여 현재 tip(~329만 블록)까지 스캔해야 한다. 모바일에서 수 시간이 소요된다.

스캔 범위 우선순위 시스템이 무엇을 먼저 스캔할지 결정한다:

PriorityValue의미
Verify6체인 reorg 감지 — 즉시 스캔
ChainTip5최신 블록 — 최근 트랜잭션 표시
FoundNote4발견된 노트의 commitment tree 완성
OpenAdjacent3계정 birthday 근처
Historic2fully-scanned height 전진
Scanned1이미 스캔됨 (건너뜀)
Ignored0관련 없음 (건너뜀)

5.2 Compact Block이란

Compact Block은 Zcash 블록의 경량화 버전이다. 라이트 클라이언트가 trial decryption을 수행하는 데 필요한 최소한의 데이터만 포함한다.

Full BlockCompact Block
크기~2MB (평균)~2KB (평균)
포함 내용모든 TX 데이터, 서명, 스크립트, 풀 메모블록 헤더 + shielded 출력의 암호화된 부분만
용도풀노드 검증라이트 클라이언트 trial decryption
대역폭100%~1% (90%+ 절약)

왜 필요한가: Shielded 트랜잭션에서 "이것이 내 트랜잭션인지" 확인하려면 모든 shielded 출력을 내 viewing key로 복호화 시도해야 한다 (trial decryption). 이 작업에 필요한 데이터는 각 출력의 암호화된 ciphertext와 ephemeral key 뿐이다. 서명, transparent 스크립트, 풀 메모 등은 불필요하다.

lightwalletd가 풀노드에서 블록을 받아 불필요한 부분을 걷어내고 compact block으로 변환하여 라이트 클라이언트에 제공한다.

Protobuf 구조:

message CompactBlock {
  uint64 height = 2;
  bytes  hash = 3;            // 블록 해시
  bytes  prevHash = 4;        // 이전 블록 해시 (체인 검증용)
  uint32 time = 5;
  repeated CompactTx vtx = 7; // 트랜잭션 목록
}

message CompactTx {
  bytes hash = 1;                             // TX 해시
  repeated CompactSaplingOutput outputs = 5;  // Sapling 출력
  repeated CompactOrchardAction actions = 6;  // Orchard 액션
}

message CompactSaplingOutput {
  bytes cmu = 1;           // note commitment
  bytes ephemeralKey = 2;  // 복호화에 필요
  bytes ciphertext = 3;    // 암호화된 노트 (앞 52바이트만)
}

ciphertext의 한계: compact block의 ciphertext는 풀 메모(512바이트)가 아니라 앞 52바이트만 포함한다. 이것으로 "내 노트인지" + "금액이 얼마인지"는 알 수 있지만, 메모 전체 내용은 알 수 없다. 메모를 읽으려면 enhancement 루프에서 풀 트랜잭션을 별도로 가져와야 한다.

5.3 In-Memory Block Source

이전에는 FsBlockDb(파일시스템 기반 캐시)를 사용했으나, 현재는 메모리 기반 MemoryBlockSource를 사용한다.

zcash_client_backend의 BlockSource trait을 구현하여, gRPC에서 받은 CompactBlock을 Vec에 저장하고 scan_cached_blocks()에 직접 전달한다. 파일 I/O가 완전히 제거되었다.

// sync_engine.rs
struct MemoryBlockSource {
    blocks: Vec<CompactBlock>,  // gRPC에서 받은 블록
}

impl BlockSource for MemoryBlockSource {
    fn with_blocks(..., mut with_block: F) {
        for block in &self.blocks {
            with_block(block.clone())?;
        }
    }
}

이로 인해 cache_path 파라미터, FsBlockDb, BlockMeta, init_blockmeta_db가 모두 불필요해졌다.

DB connection은 sync 시작 시 한 번 열리고, 모든 작업(suggest_scan_ranges, scan_cached_blocks, get_progress, run_enhancement)에서 재사용된다.

5.4 Trial Decryption & 노트 발견

zcash_client_backend의 scan_cached_blocks()가 캐시된 블록의 모든 shielded 출력에 trial decryption을 수행한다:

  • 각 Sapling 출력에 대해: DB에 있는 모든 계정의 Sapling IVK로 복호화 시도
  • 각 Orchard 액션에 대해: DB에 있는 모든 계정의 Orchard IVK로 복호화 시도
  • 복호화 성공 = 해당 계정의 노트 → 월렛 DB에 저장 (어떤 계정의 것인지 자동 판별)
  • Nullifier를 추적하여 노트가 사용되었는지 감지

멀티 계정과 Sync: Sync는 계정에 무관하게 실행된다. run_sync_inner()는 account_uuid 파라미터를 받지 않으며, DB에 등록된 모든 계정의 UFVK로 trial decryption을 수행한다. 잔액과 트랜잭션 내역 조회만 account_uuid로 특정 계정을 필터링한다.

5.5 진행률 보고

진행률은 scan-queue 기반으로 계산한다. 매 배치 후 suggest_scan_ranges()에서 남은 블록 수(remaining)와 전체 블록 수(total, Ignored 제외)를 계산하여 pct = 1.0 - remaining / total로 산출한다. 이 방식은 스캔 순서(ChainTip 우선, FoundNote, Historic)에 관계없이 정확하며, 노트가 없는 지갑에서도 올바르게 동작한다.

각 progress event에는 has_new_tx 플래그가 포함된다. ScanSummary의 received/spent note count > 0이면 true이며, Dart에서 이를 감지하여 최근 10건의 트랜잭션을 조회한다.

Dart의 stream.listen() 콜백에서 getBalance() FFI를 호출하여 balance를 갱신하고, hasNewTx || isComplete이면 getTransactionHistory(limit: 10)도 호출한다. 폴링은 없으며, 모든 상태 업데이트는 stream, EventChannel, onResume에 의해 구동된다. iOS background sync에서는 C callback → SyncProgressStreamHandler → EventChannel로 전달되며, isSyncing, isComplete, hasNewTx 등 모든 필드가 포함된다.
